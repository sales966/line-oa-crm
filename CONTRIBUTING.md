# 協作指南（CONTRIBUTING）

歡迎一起維護這個專案！這是一套**每天都在用的核心 CRM**（LINE OA 客戶進度中樞），公司同事靠它跟單、追進度、管客服。請先花幾分鐘讀完本文件，照著做，就能安全地貢獻。

本文件寫給**非資深工程背景**的同事，不需要很懂程式也能上手。有任何看不懂的地方，先問，不要硬猜著改。

---

## 0. 一句話心法

> **保守、小步、可回復。** 這是正式在用的系統，不是玩具專案。寧可慢一點、多確認一次，也不要一次大改把大家的日常工具弄壞。

---

## 1. 環境準備

第一次設定環境，照順序做：

1. **安裝 Node 22**（請用 22 版，其他版本可能有相容問題）
   - 確認版本：`node -v`，看到 `v22.x.x` 就對了。
2. **安裝後端相依套件**
   ```bash
   cd backend
   npm install
   ```
3. **建立環境設定檔**
   - 複製範本：把 `backend/.env.example` 複製一份，改名成 `backend/.env`。
   - 打開 `.env`，填入 `OPENAI_API_KEY=你的金鑰`。
   - ⚠️ 這個 `.env` **永遠不能上傳到 GitHub**（詳見第 3 節）。
4. **啟動後端**
   ```bash
   npm start
   ```
   - 預設會在 **port 4680** 執行（網址 `http://localhost:4680`）。
   - 想換 port，可在 `.env` 設 `PORT=別的號碼`。

前端（`webui/`）是**純靜態網頁**，不用另外編譯，後端啟動後直接用瀏覽器打開對應頁面即可測試。

## 2. 分支與 Pull Request 規則

**絕對不要直接把改動推到 `main`。** 一律走以下流程：

1. **先更新本機再開分支**
   ```bash
   git pull          # 推之前也要先 pull，避免衝突
   git checkout -b feature/你的功能名稱
   ```
   - 分支命名用 `feature/xxx`，例如 `feature/customer-tags`、`feature/export-fix`。
2. **在自己的分支上開發、提交**
   ```bash
   git add .
   git commit -m "簡短說明這次改了什麼"
   ```
3. **推上去並開 Pull Request（PR）**
   ```bash
   git push -u origin feature/你的功能名稱
   ```
   - 到 GitHub 上開一個 Pull Request，說明你改了什麼、為什麼改。
4. **等審閱通過後才合併**進 `main`。不要自己審自己、不要略過審閱。

> 小提醒：推之前一定 `git pull` 一次，把別人的最新改動先合進來，減少衝突。

---

## 3. 🔒 安全鐵律（最重要，務必遵守）

以下這些**絕對、絕對不能 commit 進 Git**：

| 不能上傳的東西 | 為什麼 |
| --- | --- |
| `backend/.env` | 裡面有 `OPENAI_API_KEY` 等機密金鑰 |
| `backend/data/` | 真實客戶資料庫 |
| `backend/storage/` | 客戶上傳的檔案 |
| `backend/backups/` | 資料庫備份 |
| 任何含**真實客戶名 / botId / API key / 內網 IP** 的檔案 | 洩漏客戶隱私與公司資安 |
| 內部設計文件 `PLAN.md`、`CONTRACT.md` | 屬內部文件，不對外 |

**重點觀念：**

- `.gitignore` 已經幫我們擋掉大部分上述檔案，但**人工還是要留意**——工具不是萬能的。
- **`PLAN.md` / `CONTRACT.md` 已被 gitignore，千萬不要用 `git add -f` 之類的方式強推。**
- **提交前務必看一眼 `git status`**，確認要上傳的清單裡**沒有**上面這些東西。
- 只要有一絲懷疑「這個檔案能不能上傳？」——**先不要推，先問**。金鑰一旦外洩就得全部重換，客戶資料外洩更是嚴重事故。

## 4. 改動前必做的檢查（缺一不可）

在推程式碼之前，請務必跑完以下檢查，全部通過才能開 PR：

1. **測試全綠**
   ```bash
   cd backend
   npm test
   ```
   - 目前共 **74 項測試**，必須**全部通過**。
   - 有任何一項紅（失敗），代表你可能弄壞了既有功能，先修好再說。
2. **型別檢查無誤**
   ```bash
   npx tsc --noEmit
   ```
   - 這個指令只檢查 TypeScript 型別、不會產生檔案。**沒有任何錯誤訊息**才算過。
3. **改了前端（`webui/`）要自己測**
   - 前端沒有自動測試，改完後**自己打開瀏覽器實際點一遍**，確認功能正常、沒有語法錯誤（可看瀏覽器 F12 主控台有沒有紅字）。

## 5. 穩定性原則（核心 CRM 的保命守則）

這套系統每天都有人在用，弄壞會直接影響同事工作。請遵守：

- **小步改動、保守為上。** 一個 PR 只做一件事，方便審閱與回復。
- **動資料庫 schema 或客戶資料前，先備份。**
  - 後台管理員的「**系統健康**」頁可以**立即備份**——動手前先按一下。
- **新功能盡量用隔離的資料表 / 新的路由**，不要去改動既有的「整體」流程與資料。能新增就不要改舊的。
- **`webui/customer.html` 是高風險大檔案**，結構複雜。**避免大改它的結構**，只做必要的小修改，改完務必完整自測。
- 不確定影響範圍時，先在自己分支上驗證清楚，或先在 PR 裡說明、請人一起看。

## 6. 目錄導覽（東西放在哪）

```
backend/
  src/
    routes/    ← API 路由（對外的網址端點）
    services/  ← 商業邏輯（實際做事的程式）
    db.ts      ← 資料庫 schema（資料表結構定義）
  test/        ← 自動化測試（就是 npm test 跑的東西）
webui/         ← 純靜態後台網頁（繁體中文介面）
  customer.html  ← 高風險大檔，小心改
extension-sync/ ← 插件相關（在 webui 打包）
```

- **要加新 API**：在 `backend/src/routes/` 新增路由，邏輯寫在 `backend/src/services/`。
- **要改資料表結構**：看 `backend/src/db.ts`，並記得先備份。
- **要改後台畫面**：改 `webui/` 底下的網頁。
- 瀏覽器插件是在 `webui` 打包產出的。

## 7. 用 AI 協作也要守規矩

用 AI 工具（千問 / Qwen Code CLI、通義靈碼等）幫忙寫程式**很歡迎**，但——

> **AI 改完的程式碼，一樣要走完第 4 節的檢查（`npm test` 全綠、`tsc --noEmit` 無誤、前端自測），才能推。**

AI 會犯錯、會誤刪、也可能不小心把機密寫進程式碼。**你是最後把關的人**，責任在你，不在 AI。特別注意：

- AI 產出的程式碼也要遵守第 3 節**安全鐵律**（別讓 AI 把金鑰或客戶資料寫進檔案）。
- 別讓 AI 一次大改一堆檔案就直接推——一樣**小步、可審閱**。

## 有疑問時

看不懂、不確定、覺得「這樣改好像會出事」——**先停下來問**，不要硬推。這套系統的穩定，比多趕一個功能重要得多。謝謝你的細心！
